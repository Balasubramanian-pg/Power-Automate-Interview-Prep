**What Is It?**
The Terminate action is a control connector used to immediately stop the execution of a cloud flow run and set a specific final status for that run.

**Why It Matters:**
In professional automation, not every flow should run to its natural end. The Terminate action allows developers to handle "Early Exits" (e.g., when data validation fails) or "Graceful Failures" (e.g., when a Try-Catch block catches an error), ensuring the Flow Run history accurately reflects whether the process was successful, failed, or cancelled.

**Core Concepts**

*   **Termination Status** - The final state assigned to the flow run: Succeeded, Failed, or Cancelled.
*   **Run-After Logic** - The configuration that determines if the Terminate action executes based on the success or failure of the preceding action.
*   **Immediate Halt** - Unlike a "Condition" which skips branches, Terminate stops all parallel branches and nested scopes instantly.

**Implementation**

### Basic Setup

**Terminate Action:**

```json
{
    "type": "Terminate",
    "inputs": {
        "runStatus": "Failed",
        "runError": {
            "code": "VAL-001",
            "message": "The provided Invoice ID does not match the required format."
        }
    }
}
```

> [!TIP]
> Use the "Succeeded" status for logical early exits where no further processing is needed but the flow performed its job correctly (e.g., a duplicate record was found and ignored).

### Advanced Configuration

You can use dynamic expressions to populate the Error Code and Message when the status is set to "Failed."

```javascript
// Example: Passing the actual error message from a failed action into the Terminate action
actions('Get_File_Metadata')?['outputs']?['body']?['message']
```

> [!WARNING]
> If you place a Terminate action inside a loop (Apply to Each), it will stop the **entire flow**, not just the current iteration. To stop only the current iteration, use the "Control - Continue" or "Control - Break" logic (if available) or a conditional wrapper.

**Common Patterns**

### Pattern 1: Validation Guard Clause

```
Trigger: When an item is created
|
Condition: Is 'Amount' greater than 0?
|-- No: Terminate (Status: Succeeded)
|-- Yes: Continue processing...
```

This pattern prevents the flow from wasting API calls or resources on invalid data without marking the run as a "Failure" in the history.

### Pattern 2: Try-Catch Error Handling

```
Scope: Try
|-- Action: Update Database
|
Scope: Catch (Run-after: Has Failed)
|-- Action: Send Admin Alert
|-- Action: Terminate (Status: Failed, Code: DB_ERR, Message: 'Update failed')
```

This ensures that even though you "handled" the error with a Catch scope, the overall flow run is still marked as "Failed" for monitoring and analytics purposes.

**Performance & Limits**

> [!IMPORTANT]
> **Execution Limits:**
> - Terminate is a standard action and counts toward daily request limits.
> - **Scope:** It terminates the entire flow run, including all active parallel branches.
> - **License:** Available on all Power Automate plans (Standard).

**Optimization Tips:**
- Use Terminate to save on "Action Executions" by exiting early before reaching expensive connectors.
- Always provide a descriptive "Message" when using the Failed status to assist in debugging from the Run History view.
- Use "Cancelled" status specifically for manual interventions or timeouts.

**When NOT to use:**
- Do not use Terminate if you want parallel branches to continue running.
- Do not use Terminate inside a loop if you only want to skip one item; use a Condition instead.

**Interview Questions**

### Beginner

**Q:** What are the three statuses you can set using the Terminate action?

<details>
<summary>Answer</summary>

The three statuses are:
1.  **Succeeded:** The flow stops and is marked as a success (Green).
2.  **Failed:** The flow stops and is marked as a failure (Red). This requires a custom Code and Message.
3.  **Cancelled:** The flow stops and is marked as cancelled (Grey).

Example: Use "Succeeded" when a business rule determines no further action is needed, but no error occurred.

</details>

### Intermediate

**Q:** If you have two parallel branches and one branch hits a Terminate action, what happens to the other branch?

<details>
<summary>Approach</summary>

**Solution:**
The other branch is immediately stopped. The Terminate action is "Global" to the flow run. It does not matter how deep the action is nested (inside Scopes, Loops, or Conditions); it kills the entire execution thread.

> [!TIP]
> If you need to stop one branch but let others continue, you should use a "Condition" or "Switch" to wrap the logic rather than a Terminate action.

**Follow-up they'll ask:** "How can I pass a specific error message to the parent flow if I'm using a Child Flow?"  
**Answer:** Use the Terminate action with the "Failed" status in the Child Flow. The Parent Flow will see the Child Flow action as "Failed" and can capture the error message.

</details>

### Advanced

**Q:** Explain the architectural significance of using Terminate (Failed) inside a Catch scope versus letting the flow fail naturally.

<details>
<summary>Solution</summary>

**Architecture:**
Using an explicit Terminate action allows for **Clean State Reporting**. 

**Key considerations:**
- **Natural Failure:** If a flow fails naturally, the error message is often a raw JSON/API error which is difficult for business users to read.
- **Explicit Terminate:** By catching the error and using Terminate, you can transform a technical error (e.g., "404 Not Found") into a business-friendly error (e.g., "Customer Record not found in CRM").

**Trade-offs to discuss:**
- **Option A (Terminate):** Pros: Custom error codes, cleaner analytics. Cons: Requires more actions/complexity in the flow design.
- **Option B (Natural Failure):** Pros: Simple to build. Cons: Harder to monitor and troubleshoot at scale; "Catch" blocks won't trigger if you don't have them, but if you do have them, the flow will mark as "Succeeded" unless you explicitly Terminate.

</details>

**Comparison Table**

| Status | Run History Color | Required Inputs | Typical Use Case |
|----------|-----------|-----------|---------------|
| **Succeeded** | Green | None | Logical early exit (e.g., filter out unwanted data). |
| **Failed** | Red | Code, Message | Error handling in Try-Catch blocks. |
| **Cancelled** | Grey | None | User-initiated stop or timeout logic. |

**Real Example**

> [!NOTE]
> **Problem:** An HR Onboarding flow was failing because new hires occasionally lacked a personal email address. The flow would error out at the "Send Email" step, making it look like a system crash.  
> **Solution:** Added a condition to check for the email. If null, the flow uses a Terminate action with status "Succeeded" and a note "Skipped: No email provided."  
> **Result:** The IT team stopped receiving "Flow Failed" alerts for expected data gaps.

**Practice**

**Exercise 1:** Create a flow that triggers when a file is uploaded to OneDrive. If the file size is greater than 5MB, terminate the flow as "Failed" with the message "File too large." Otherwise, terminate as "Succeeded."

<details>
<summary>Hint</summary>

Use the "Get file metadata" action to find the file size, then use a "Condition" action.

</details>

<details>
<summary>Solution</summary>

1. Trigger: **When a file is created**
2. Action: **Get file metadata**
3. Condition: `outputs('Get_file_metadata')?['body/Size']` is greater than `5242880`
4. If Yes: **Terminate** (Status: Failed, Code: `SIZE_LIMIT`, Message: `File exceeds 5MB limit`)
5. If No: **Terminate** (Status: Succeeded)

**Key points:**
```javascript
// Size is usually in bytes
5 * 1024 * 1024 = 5242880
```

</details>

**Troubleshooting Guide**

> [!CAUTION]
> **Common Errors:**

| Error | Cause | Fix |
|-------|-------|-----|
| Flow marked "Succeeded" despite error | A Catch scope handled the error but did not include a Terminate action. | Add a Terminate (Failed) action at the end of your Catch scope. |
| Terminate action skipped | The "Run After" settings of the Terminate action are not configured to handle the preceding action's state. | Check "Configure Run After" and ensure "Has Failed" or "Is Skipped" is checked if necessary. |
| Parallel branch stopped unexpectedly | One branch hit a Terminate action, killing the entire run. | Replace Terminate with a "Condition" to wrap the logic you want to skip. |
